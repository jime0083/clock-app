## Problem 9: バックグラウンド・ロック画面でアプリのアラーム音が鳴らない (2026-04-13)

### 症状
- **アプリ起動時（フォアグラウンド）**: アラーム音が鳴る ✓
- **ホーム画面（バックグラウンド）**: 通知音は鳴るがアプリのアラーム音が鳴らない ✗
- **ロック画面**: 通知音は鳴るがアプリのアラーム音が鳴らない ✗

### 重要な区別
- **通知音**: FCM/APNsが再生する短い音（"default" = 約2秒）→ **動作している** ✓
- **アラーム音**: `audioService.playAlarmSound()`が再生するループ音（アラーム音.m4a）→ **バックグラウンドでは動作しない** ✗

### 調査で排除した原因
- ✗ Cloud Functionsの問題 → ログで「Successfully sent alarm」を確認
- ✗ FCM送信の問題 → 通知は正常に届いている
- ✗ iOS通知設定 → 全てオン
- ✗ サイレントスイッチ → オフ
- ✗ `setBackgroundMessageHandler` → 無効化しても改善せず
- ✗ FCMペイロード形式 → 複数パターン試行済み

### 根本原因（2026-04-15 特定）

**iOSのバックグラウンド制限により、アプリがバックグラウンドにある時はJavaScriptコードが実行されない**

現在のアーキテクチャ:
```
フォアグラウンド時:
  FCM通知受信 → setForegroundMessageHandler → alarmService.triggerAlarmFromFCM()
    → audioService.playAlarmSound() → アラーム音がループ再生 ✓

バックグラウンド時:
  FCM通知受信 → iOS が通知を表示（通知音 "default" が鳴る）
    → JSコード実行されない → audioService.playAlarmSound() 呼ばれない ✗
```

**コードの流れ**:
1. `alarmService.ts:282` - `triggerAlarmFromFCM()` → `handleAlarmTriggered()`
2. `alarmService.ts:186` - `audioService.playAlarmSound(customSound, true)`
3. `audioService.ts:36-95` - expo-audioで`アラーム音.m4a`をループ再生

**問題**: バックグラウンドでは上記のJSコードが実行されないため、アラーム音が鳴らない

### 解決策

**APNs通知のカスタムサウンドとして、アラーム音をバンドルに含め、FCMペイロードで指定する**

iOSではAPNs通知のサウンドは最大30秒まで指定可能。カスタムアラーム音をアプリにバンドルし、FCMペイロードの`sound`フィールドで指定することで、バックグラウンドでもアラーム音が鳴るようにする。

```javascript
// Firebase Functions - FCMペイロード
apns: {
  payload: {
    aps: {
      alert: { title: "起床時間です！", body: "..." },
      sound: "alarm.caf",  // カスタムアラーム音（30秒以内）
      "interruption-level": "time-sensitive",
    },
  },
},
```

### 実装タスク

1. **アラーム音ファイルの準備**
   - `アラーム音.m4a`を30秒以内にトリミング
   - CAF形式に変換（`alarm.caf`）

2. **iOSバンドルへの追加**
   - Expo config pluginでXcodeプロジェクトに追加
   - または手動で`ios/okiroya/`に配置

3. **FCMペイロードの更新**
   - `sound: "alarm.caf"`に変更

4. **ユーザー起動時のアラーム継続**
   - 通知タップでアプリ起動 → ループアラーム再生開始
   - スクワット完了まで継続

#### 2. Firebase Functionsのデプロイ失敗 [重大度: 高]
- **問題**:
  - `sound: "default"` への変更をデプロイできていない
  - Cloud Buildで「An unexpected error occurred」エラーが発生
  - 古いコード（sound: "alarm.caf"）が本番で動作中
- **確認方法**: Google Cloud Console > Cloud Build でエラーログを確認

#### 3. prebuild時にカスタムファイルが消える [重大度: 中]

  - `npx expo prebuild --clean` で `ios/` フォルダが再生成される
  - 手動で追加した `alarm.caf` が消える

### 根本原因の特定

**バックグラウンドでアラーム音が鳴らない直接の原因**:
- FCM通知のペイロードで指定された `alarm.caf` がアプリバンドルに存在しない
- iOSは存在しないサウンドファイルを無視し、無音で通知を表示する

**解決策の優先順位**:
1. Firebase Functionsをデプロイして `sound: "default"` を適用（最も簡単）
2. または `alarm.caf` をアプリバンドルに正しく含める（Expo config plugin必要）

## Problem 4: バックグラウンドで通知音が鳴らない（デプロイ後も継続）(2026-04-10)

### 症状
- Firebase Functionsデプロイ成功後もバックグラウンドで通知音が鳴らない
- 通知自体は正常に表示される
- Xcodeコンソールには`appEntersBackground`イベントが表示されている

### 調査結果

#### 確認済みの設定
- `ios/app/Info.plist`: UIBackgroundModes に `remote-notification`, `audio`, `fetch` が設定済み ✓
- `app.config.ts`: infoPlist に UIBackgroundModes が設定済み ✓
- `functions/src/index.ts`: `sound: "default"` に変更済み・デプロイ済み ✓
- `notificationService.ts`: `interruptionLevel: 'timeSensitive'` が設定済み ✓

#### 根本原因の特定

**原因1: iOSのサイレントスイッチ（物理スイッチ）[重大度: 致命的・可能性高]**
- iPhoneの側面の物理スイッチがオレンジ色（サイレントモード）の場合、通常の通知音は鳴らない
- `sound: "default"` はサイレントスイッチを**無視しない**
- `interruptionLevel: 'timeSensitive'` もサイレントスイッチを**無視しない**
- **サイレントスイッチを無視して音を鳴らすには Critical Alert が必要**（Apple特別承認必要）

**原因2: iOS通知設定でサウンドがオフ [重大度: 高・可能性中]**
- 設定 > 通知 > オキロヤ > サウンドがオフになっている可能性
- ユーザーが意図せずオフにしている場合がある

**原因3: 着信音/通知音の音量がゼロ [重大度: 高・可能性中]**
- 設定 > サウンドと触覚 > 着信音と通知音がゼロになっている可能性

**原因4: バックグラウンドでJSコードが実行されない [重大度: 情報]**
- iOSではバックグラウンドでJavaScriptランタイムが停止
- `audioService.playAlarmSound()` はバックグラウンドでは実行されない
- → FCM通知の`sound`設定のみに依存する必要がある（現状はこれで正しい）

### 結論

**最も可能性が高い原因**: ユーザーのiPhoneがサイレントモード（物理スイッチがオレンジ色）

**解決策の選択肢**:
1. **ユーザーにサイレントモードをオフにしてもらう**（即座に解決・制限あり）
2. **Critical Alert エンタイトルメントを取得**（Apple承認必要・時間がかかる）
3. **アプリ内でユーザーに注意喚起**（サイレントモードでは音が鳴らない旨を表示）

## Problem 10: バックグラウンド復帰・アプリ終了後起動でスクワット画面が表示されない (2026-04-16) [解決済み 2026-04-18]

### 症状
1. **バックグラウンドから復帰**: アラーム5分以内にバックグラウンドから復帰してもスクワット画面が表示されない
2. **アプリ終了後に起動**: アラームが鳴った後、アプリを終了状態から起動してもスクワット画面が表示されない

### 正常に動作するケース
- 通知タップでアプリを開く → スクワット画面が表示される ✓
- アプリアイコンタップで開く（通知無視）→ スクワット画面が表示される ✓

### 根本原因（2026-04-18 特定）

**二重AppStateリスナーと競合状態**

1. **二重リスナー問題**:
   - `alarmService.ts` (line 121-126): 内部で`AppState.addEventListener`を設定
   - `App.tsx` (line 90-120): 独自にも`AppState.addEventListener`を設定
   - 同じイベントに対して2つのリスナーが存在

2. **競合フロー**:
   ```
   AppState → 'active'
       ↓
   alarmService.handleAppStateChange() [リスナー1]
       ↓ checkAndStartAlarmIfNeeded()
       ↓ handleAlarmTriggered()
       ↓ this.alarmState = 'ringing'
       ↓ this.onAlarmTriggeredCallback() → setIsAlarmRinging(true)
       ↓
   App.tsx handleAppStateChange() [リスナー2]
       ↓ alarmService.checkAndStartAlarmIfNeeded()
       ↓ handleAlarmTriggered() → "Already ringing, skipping" (line 168-171)
       ↓ コールバック呼び出しがスキップされる
   ```

3. **コールバックStale Closure問題**:
   - `alarmService.setOnAlarmTriggered()` で設定されるコールバックは初期化時の参照
   - React状態の更新がバッチ処理で遅延される可能性
   - コールバック内の`setIsAlarmRinging`がstale closureの影響を受ける可能性

### 解決策

1. **alarmServiceの内部AppStateリスナーを削除**: App.tsxで一元管理
2. **handleAlarmTriggeredの改善**: ringing状態でもコールバックを呼び出す（UI同期のため）
3. **App.tsxのAppStateリスナーを主要な制御ポイントに**: 直接状態更新を行い、コールバックへの依存を減らす
4. **競合防止**: alarmServiceからcheckAlarmWindow()を呼び出し、App.tsx側でsetIsAlarmRinging(true)を直接設定

### 実装内容（2026-04-18）

**alarmService.ts**:
- `setupAppStateListener()` を削除（App.tsxに一元化）
- `handleAppStateChange` を削除
- `handleAlarmTriggered()` で `alarmState === 'ringing'` の場合もコールバックを呼び出すよう変更

### 技術的背景

**HealthKit連携に必要な3つの要素:**

1. **Entitlements** (`app.entitlements`): アプリがHealthKitを使用する権利を宣言
2. **Capability** (Xcode設定): Xcodeプロジェクトでの有効化（自動署名に必要）
3. **Usage Description** (`Info.plist`): ユーザーに表示する権限要求の説明文

現在の問題は、1と3は設定済みだが、**2のXcodeでのケイパビリティ設定が欠けている**可能性が高い。

Expo prebuildはentitlementsファイルとInfo.plistを生成するが、Xcodeの "Signing & Capabilities" への自動追加は保証されない場合がある。

### 解決（2026-04-22）

**根本原因の追加特定:**
- `react-native-health` がNew Architecture（Fabric/TurboModules）に対応していない
- `react-native-reanimated` v4.x がNew Architectureを必須化しているため、両ライブラリが共存不可能
- Xcodeコンソールに `[TypeError: undefined is not a function]` エラーが発生

**解決策:**
1. `react-native-health` をアンインストール
2. `@kingstinct/react-native-healthkit` をインストール（Nitro Modules対応でNew Architecture互換）
3. `react-native-nitro-modules` を依存関係として追加
4. `healthKitService.ts` を新APIで全面書き換え
5. `app.config.ts` でプラグインを更新
6. XcodeでHealthKitケイパビリティを手動追加（"Clinical Health Records"はチェックしない）

**結果:**
- ✅ 初回起動時にHealthKit権限ダイアログが正常に表示
- ✅ スクワット完了後、ヘルスケアアプリに「機能的筋力トレーニング」が記録される

**学び:**
- New Architectureを使用するプロジェクトでは、ネイティブモジュールのNew Architecture対応を確認する必要がある
- `@kingstinct/react-native-healthkit` はNitro Modulesを使用しており、New Architectureとの互換性が高い
